## 0. Purpose & gate
Prove the WHOLE parallel custom-layout path against the REAL Rectangle source before any
model/UI code — not just "a shortcut fires." This doc grounds every step in confirmed APIs
(finding #7: no guessed class names) and defines a runnable harness whose **7 checks must all
pass** to clear M0.5. Output: `SPIKE.md` recording every confirmed API, the canonical hotkey
serialization, and the coordinate helper used.

## 1. Confirmed APIs (ground truth, read from source this milestone)
All paths under `/Users/projas/divvy-2`. Verified by direct read.

### 1.1 Apply path (AX frame-set) — §3.7
- Focused window: `AccessibilityElement.getFrontWindowElement() -> AccessibilityElement?`
  — `Rectangle/AccessibilityElement.swift:299`.
- **Direct setter already exists** (no need to add one): `func setFrame(_ frame: CGRect,
  adjustSizeFirst: Bool = true)` — `AccessibilityElement.swift:124`. Internally sets `size`,
  then `position`, then `size` (AX requires separate set; comment at :121), and toggles
  `AXEnhancedUserInterface` around the resize.
- Read-back: `var frame: CGRect` (`AccessibilityElement.swift:116`) built from settable
  `position`/`size`. Used for assertions.
- This path is what `StandardWindowMover.moveWindow` itself calls
  (`Rectangle/WindowMover/StandardWindowMover.swift:9`): `windowElement.setFrame(rect.screenFlipped)`.
  It does **not** route through `WindowAction`/`WindowManager.execute`.

### 1.2 Coordinate conversion — §3.8
- `CGRect.screenFlipped` / `CGPoint.screenFlipped` — `Rectangle/Utilities/CGExtension.swift:5`.
  Flips Y about **`NSScreen.screens[0].frame.maxY`** (the PRIMARY screen height), exactly the
  §3.8 requirement. AppKit (bottom-left) ↔ AX (top-left, primary-referenced).
- Primary screen == `NSScreen.screens[0]` == `NSScreen.main` (`CGExtension.swift:7`,
  `ScreenDetection.swift:181`).

### 1.3 Screen selection — §3.4
- `ScreenDetection().detectScreens(using frontmostWindowElement:) -> UsableScreens?`
  — `Rectangle/ScreenDetection.swift:7`. `UsableScreens` carries `currentScreen: NSScreen`,
  `frameOfCurrentScreen`, `screensOrdered`, `numScreens`.
- Rule: `screenContaining(_:screens:)` (`ScreenDetection.swift:48`) = **largest-intersection
  percentage**, early-return on full containment; fallback `NSScreen.main`. Matches §3.4
  verbatim — adopt as-is.
- Frame basis: custom layouts use **raw `currentScreen.visibleFrame`** (gap = 0 per §3.5).
  Do NOT call `adjustedVisibleFrame(...)` (`ScreenDetection.swift:153`), which subtracts
  gaps/Stage/Todo/notch — that is Rectangle's own gap system we are deliberately not touching.
  (`visibleFrame` already excludes menu bar / Dock, incl. notch safe-area.)

### 1.4 Hotkey registration — §3.3
- Use **`MASShortcutMonitor.shared()`** directly (Rectangle's own `ShortcutManager` instead uses
  the higher-level `MASShortcutBinder`; our parallel system keys by layout `UUID`, so the lower
  `MASShortcutMonitor` is the right seam). Confirmed API (MASShortcut SPM checkout headers,
  `MASShortcutMonitor.h`):
  - `+ (instancetype) sharedMonitor;` → Swift `MASShortcutMonitor.shared()`
  - `- (BOOL) registerShortcut:(MASShortcut*)s withAction:(dispatch_block_t)a;`
    **returns Bool** — handles rev3 #3's "actual registration-failure return."
  - `- (void) unregisterShortcut:(MASShortcut*)s;` / `unregisterAllShortcuts`
- `MASShortcut` canonical fields: `keyCode: NSInteger`, `modifierFlags: NSEventModifierFlags`
  (`MASShortcut.h`); `init(keyCode:modifierFlags:)`.

### 1.5 Lifecycle signals — §3.3 / rev3 #1
- Recording-in-progress: `Notification.Name.shortcutRecording`
  (`Rectangle/Utilities/NotificationExtension.swift:20`), posted with a `Bool` payload by
  `Rectangle/PrefsWindow/ShortcutRecordingObserver.swift:61`. Rectangle's `ShortcutManager`
  guards `shortcutsSuspendedForRecording` (`ShortcutManager.swift:12,39,178`).
- App-ignore / global-disable: `ApplicationToggle.shortcutsDisabled` (static Bool,
  `Rectangle/ApplicationToggle.swift:10`), flipped on front-app change via
  `disableShortcuts()`/`enableShortcuts()` (:50–68), which also post
  `Notification.Name.windowSnapping` (Bool).

### 1.6 Conflict model — §3.3 / rev3 #3
- Precedent: `TodoManager.conflict(for:ignoringTodoDefaultsKey:userDefaults:) -> TodoShortcutConflict?`
  (`Rectangle/TodoMode/TodoManager.swift:284`). It compares `ShortcutCycle.ShortcutIdentity(shortcut)`
  against every `WindowAction.active` shortcut (via `ShortcutCycle.shortcut(for:userDefaults:)`)
  and the other todo keys. **We mirror this**: reject a chord whose `ShortcutIdentity` matches any
  live `WindowAction` shortcut OR any other custom layout.
- Window Actions themselves use `PassthroughShortcutValidator` (`PrefsViewController.swift:145`),
  i.e. Rectangle does NOT reject conflicts for its own actions — so OUR system owns the conflict gate.

### 1.7 History / restore — §3.6
- `AppDelegate.windowHistory: WindowHistory` (`AppDelegate.swift:15`). `WindowHistory` is two
  dicts: `restoreRects [CGWindowID: CGRect]` and `lastRectangleActions [CGWindowID: RectangleAction]`
  (`Rectangle/WindowHistory.swift`). Only `WindowManager.execute(...)` writes them. Our direct
  `setFrame` path never calls that ⇒ structural opt-out; provable by before/after snapshot.

### 1.8 WindowAction is a fixed enum (BLOCKER confirmed)
- `enum WindowAction: Int, Codable` with hardcoded `static let active` (`WindowAction.swift:13,141`);
  execution is notification-driven (`post()` → `Notification.Name(action.name)` →
  `WindowManager.execute`). Cannot add runtime cases ⇒ parallel subsystem is required (as planned).

## 2. Canonical `HotkeyData` (resolves §3.2 rev2 #3 / rev3 #4)
Rectangle already ships the exact Codable shape we need — reuse its design rather than inventing:
`struct Shortcut: Codable { let keyCode: Int; let modifierFlags: UInt }` with
`init(masShortcut:)` and `toMASSHortcut()` (`WindowAction.swift:1182`). The on-disk MASShortcut
representation is the `MASDictionaryTransformer` dict `{ "keyCode": Int, "modifierFlags": Int }`.

**Decision:** `HotkeyData` is a Codable struct `{ keyCode: Int, modifierFlags: UInt, schemaVersion: Int = 1 }`
mirroring `Shortcut`, with `init(_ MASShortcut)` and `func toMASShortcut()`. Persisted inside each
`CustomLayout` in the JSON array under our own key **`com.perg593.divvy2.customLayouts`**. Migration:
`schemaVersion` starts at 1; the spike commits this representation into `SPIKE.md` and asserts a
MASShortcut → HotkeyData → MASShortcut round-trip preserves `keyCode`/`modifierFlags`.

## 3. Harness design (minimal upstream coupling — R2)
- **New, isolated code** under `Rectangle/Divvy2Spike/` (one new directory; nothing edited in
  existing files except a single guarded call). Files:
  - `Divvy2SpikeRunner.swift` — orchestrates the 7 checks, writes `SPIKE.md`, logs `PASS`/`FAIL`
    and a final summary line; sets process exit code non-zero on any failure (CI-style gate).
  - `Divvy2SpikeCustomLayout.swift` — the *minimal* spike types: `HotkeyData`, a `CustomLayout`
    stub, and a `SpikeCustomLayoutShortcutManager` (register/unregister via `MASShortcutMonitor`,
    conflict check, lifecycle suspension). These are spike-grade, NOT the final M1/M2 types.
- **Trigger:** in `AppDelegate.applicationDidFinishLaunching`, ONE guarded line:
  `if CommandLine.arguments.contains("--divvy2-spike") { Divvy2SpikeRunner.run() }`. No other
  production code paths change. Normal launches are unaffected.
- **Invocation for the gate:** run the built binary directly so stdout is captured:
  `"…/Moving Van.app/Contents/MacOS/Moving Van" --divvy2-spike` (same bundle id/signature ⇒ keeps
  the AX grant). Output path: `SPIKE.md` written to `$DIVVY2_SPIKE_OUT` (default the repo root).
- **Window-move target (deterministic):** the spike creates its OWN `NSWindow`
  (`setActivationPolicy(.regular)` + `activate` + `makeKeyAndOrderFront`), then fetches it via
  `getFrontWindowElement()` and moves it through the §3.4/§3.8 matrix. Self-contained and
  repeatable (no dependence on the user's other windows). A best-effort SECONDARY cross-app move
  (launch TextEdit, move its window) is attempted for fidelity but is non-gating; its
  availability is recorded either way. Activation policy is restored to `.accessory` at the end.

## 4. The seven gated checks (each → a real API + an assertion in SPIKE.md)
1. **Register + reload (rev2 #1).** Register two hardcoded chords via
   `MASShortcutMonitor.shared().registerShortcut(_:withAction:)`; assert both return `true` and
   their actions fire (set a flag). Then `unregisterShortcut` one and re-register it from a
   simulated store-change; assert the manager retains/rebinds correctly (no leak, action still
   fires exactly once). Proves unregister + manager lifetime/retention.
2. **Conflict rejection (rev3 #3).** Inject a known `WindowAction` shortcut (e.g. `leftHalf` =
   ⌃⌥←) into UserDefaults via the `MASDictionaryTransformer`; attempt to bind a custom layout with
   the SAME chord → assert **rejected** (conflict matches a live WindowAction). Attempt a duplicate
   **custom** chord → assert **rejected**. Bind a non-conflicting chord → assert **registered**
   (and `registerShortcut` returned `true`). Record the monitor's Bool return for each. Clean up
   the injected default afterward.
3. **Lifecycle suppression (rev3 #1).** Post `Notification.Name.shortcutRecording` (object `true`)
   → assert a fired custom chord does NOT execute (suspended). Post `false` → resumes. Set
   `ApplicationToggle.shortcutsDisabled = true` (and restore) → assert custom chord is suppressed
   while disabled. Proves the manager honors the SAME signals as `ShortcutManager`.
4. **Real window move + no history writes (rev2 #2).** Snapshot
   `AppDelegate.windowHistory.restoreRects` and `.lastRectangleActions` (counts + contents); move
   the spike window via `getFrontWindowElement().setFrame(targetRectAppKit.screenFlipped)`; assert
   the read-back `frame` equals the target within rounding tolerance AND both history dicts are
   **unchanged**. Repeat the same chord twice → assert idempotent (no drift).
5. **Coordinate contract (rev3 #2).** Validate §3.8 across: full-screen-ish, **top-half**,
   **bottom-half** (catches the Y-flip), and secondary display if attached. For each, map a
   `NormalizedRect` onto `currentScreen.visibleFrame`, apply, read back, assert match within ≤1px.
   **Notched-display fallback (rev3 #10):** if no notched/secondary display is present, instead
   assert menu-bar/Dock `visibleFrame` behavior on the primary display and record the
   unavailable-hardware case in `SPIKE.md`; the gate still passes.
6. **Hotkey serialization (rev2 #3 / rev3 #4).** Build a `MASShortcut`, convert to `HotkeyData`,
   JSON round-trip it, convert back; assert `keyCode`/`modifierFlags` preserved. Commit the ONE
   concrete `HotkeyData` representation + the `schemaVersion`/migration note into `SPIKE.md`.
7. **API confirmation (finding #7).** Record into `SPIKE.md` the confirmed names/signatures for
   `WindowAction`, `WindowManager`/`WindowCalculation*` (the path we bypass), `ScreenDetection`,
   `AccessibilityElement`, `ShortcutManager`, `PrefsViewController` + `MASShortcutView` (the real
   recorder), and `MASShortcutMonitor` — as enumerated in §1 above.

**Gate:** all seven PASS (check 5's secondary-display sub-case may be recorded as unavailable
without failing). The runner's final line is `SPIKE RESULT: PASS|FAIL (n/7)`.

## 5. Risks / unknowns to resolve during the spike
- **R-spike-1 Own-window AX fidelity.** Moving our own process's window via AX should behave like a
  cross-app move, but `AXEnhancedUserInterface` and async AX settling differ per app. Mitigation:
  read-back with a short retry/settle loop and a ≤1px tolerance; the best-effort TextEdit move
  cross-checks fidelity.
- **R-spike-2 LSUIElement activation.** Temporarily switching activation policy to `.regular` to
  own a key window may briefly show a Dock icon. Acceptable for a dev spike; restored at the end.
- **R-spike-3 MASShortcutMonitor vs MASShortcutBinder coexistence.** Rectangle binds via
  `MASShortcutBinder` (which itself drives `MASShortcutMonitor.shared()`). Registering our own
  shortcuts on the same shared monitor must not disturb Rectangle's bindings. The spike registers
  only novel chords and unregisters them at teardown; it asserts Rectangle's own bindings still
  fire afterward.
- **R-spike-4 Injecting a WindowAction default.** Check 2 writes a real UserDefaults shortcut for a
  WindowAction; must be removed in teardown so it doesn't leak into the user's prefs. The spike
  snapshots and restores the affected keys.
- **R-spike-5 Writing SPIKE.md from the app.** The app is not sandboxed; writing to the repo path
  is fine, but the path is dev-specific. Use `$DIVVY2_SPIKE_OUT` with a repo-root default and log
  the path.

## 6. Out of scope for M0.5 (deferred to M1+)
Real `CustomLayoutStore` persistence/export-import, the production
`CustomLayoutShortcutManager`, Preferences UI (the real `MASShortcutView` add/edit/delete), gaps,
and per-app rules. M0.5 builds only throwaway spike-grade types to PROVE the path.
