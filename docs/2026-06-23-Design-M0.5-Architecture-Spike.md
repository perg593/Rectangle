Changelog: rev1 → rev2 fixes a BLOCKER (can't write `ApplicationToggle.shortcutsDisabled`),
and MAJORs: primary-screen ≠ `NSScreen.main`; circular coordinate assertion; own-window move
false-greening the cross-app path; live-`UserDefaults` leakage; `MASShortcutMonitor` chord-scoped
unregister ownership; and a too-weak serialization proof. rev2 → rev3 hardens two teardown paths:
the `disableApp` lifecycle check now requires an initially-ENABLED state + a unique fake bundle id +
`reloadFromDefaults()` and asserts the flag returns to its initial value; the cross-app target
requires a FRESH verified TextEdit PID (abort if already running); and the spike hook is moved to run
after `accessibilityTrusted()` so `applicationToggle` is non-nil. rev3 → rev4 removes the last
real-user-state hazards: the cross-app window-move target is now a purpose-built DISPOSABLE helper app
(no documents, no saved-state restoration) instead of TextEdit; check 1's coexistence proof is now
NON-mutating (no Rectangle action is ever fired against a real window); and crash-cleanup uses
`DispatchSourceSignal` on the main queue, not a raw POSIX signal handler. Inline `C1#n`/`C2#n`/`C3#n`
tags map clauses to the Codex round they answer.

## 0. Purpose & gate
Prove the WHOLE parallel custom-layout path against the REAL Rectangle source before any
model/UI code — not just "a shortcut fires." This doc grounds every step in confirmed APIs
(finding #7: no guessed class names) and defines a runnable harness whose **7 gated checks must
all PASS** to clear M0.5. Each check asserts against **independent ground truth** wherever a
self-confirming assertion is possible (C1#circular). Output: `SPIKE.md` recording every confirmed
API, the canonical hotkey serialization, and the coordinate helper used.

## 1. Confirmed APIs (ground truth, read from source this milestone)
All paths under `/Users/projas/divvy-2`. Verified by direct read.

### 1.1 Apply path (AX frame-set) — §3.7
- Focused window: `AccessibilityElement.getFrontWindowElement() -> AccessibilityElement?`
  — `Rectangle/AccessibilityElement.swift:299`.
- **Direct setter already exists** (no need to add one): `func setFrame(_ frame: CGRect,
  adjustSizeFirst: Bool = true)` — `AccessibilityElement.swift:124`. Internally sets `size`,
  `position`, `size` (AX requires separate set; comment at :121) and toggles
  `AXEnhancedUserInterface` around the resize. **Side-effect note (C1#enhancedUI):** the
  enhanced-UI toggle can make the *immediate* read-back lag; assertions use a short settle/retry
  loop (≤300 ms, polling `frame`) before comparing.
- Read-back: `var frame: CGRect` (`AccessibilityElement.swift:116`) from settable `position`/`size`.
- This path is exactly what `StandardWindowMover.moveWindow` calls
  (`Rectangle/WindowMover/StandardWindowMover.swift:9`); it does **not** route through
  `WindowAction`/`WindowManager.execute`.

### 1.2 Coordinate conversion — §3.8 (C1#primaryScreen)
- `CGRect.screenFlipped` / `CGPoint.screenFlipped` — `Rectangle/Utilities/CGExtension.swift:5`.
  Flips Y about **`NSScreen.screens[0].frame.maxY`** — i.e. the FIRST screen in `NSScreen.screens`,
  which is Rectangle's conversion basis.
- **Correction:** `NSScreen.screens[0]` is NOT proven equal to `NSScreen.main`. `NSScreen.main` is
  the screen with the key window and can change; `ScreenDetection.swift:181` itself compares against
  `NSScreen.screens.first`, not `.main`. Our own coordinate helpers MUST flip about
  `NSScreen.screens[0]` and MUST NOT use `NSScreen.main` (except where `ScreenDetection` already
  uses it as a documented fallback). The spike records the actual `screens[0]` vs `.main` identity
  observed at run time.

### 1.3 Screen selection — §3.4
- `ScreenDetection().detectScreens(using frontmostWindowElement:) -> UsableScreens?`
  — `Rectangle/ScreenDetection.swift:7`. `UsableScreens` carries `currentScreen: NSScreen`,
  `frameOfCurrentScreen`, `screensOrdered`, `numScreens`.
- Rule: `screenContaining(_:screens:)` (`ScreenDetection.swift:48`) = **largest-intersection
  percentage**, early-return on full containment; fallback `NSScreen.main`. Adopt verbatim.
- Frame basis: custom layouts use **raw `currentScreen.visibleFrame`** (gap = 0 per §3.5).
  Do NOT call `adjustedVisibleFrame(...)` (`ScreenDetection.swift:153`), which subtracts
  gaps/Stage/Todo/notch — Rectangle's own gap system we deliberately don't touch. (`visibleFrame`
  already excludes menu bar / Dock incl. notch safe-area.)

### 1.4 Hotkey registration + OWNERSHIP — §3.3 (C1#ownership)
- Use **`MASShortcutMonitor.shared()`** directly (Rectangle's `ShortcutManager` uses the higher
  `MASShortcutBinder`, which itself drives the SAME `MASShortcutMonitor.shared()`). Confirmed API
  (MASShortcut SPM checkout headers):
  - `+ sharedMonitor` → Swift `MASShortcutMonitor.shared()`
  - `- (BOOL) registerShortcut:(MASShortcut*)s withAction:(dispatch_block_t)a;` **returns Bool**.
  - `- (void) unregisterShortcut:(MASShortcut*)s;` / `unregisterAllShortcuts`.
- **Ownership rule (critical):** `unregisterShortcut(_:)` is **chord-scoped, not owner-scoped** —
  it removes the monitor entry for that chord regardless of who registered it (binder uses the same
  shared monitor; it removes by shortcut key). Therefore the custom manager:
  1. runs the conflict check (1.6) BEFORE attempting `registerShortcut`;
  2. tracks ONLY its own successfully-registered, non-conflicting chords (a `[UUID: MASShortcut]`);
  3. NEVER calls `unregisterShortcut` for a rejected/failed/duplicate chord (so it can never tear
     down a chord Rectangle's binder owns);
  4. on teardown unregisters only its tracked owned chords.
- `MASShortcut` canonical fields: `keyCode: NSInteger`, `modifierFlags: NSEventModifierFlags`;
  `init(keyCode:modifierFlags:)`.

### 1.5 Lifecycle signals — §3.3 / rev3 #1 (C1#blockerDisabled)
- Recording-in-progress: `Notification.Name.shortcutRecording`
  (`Rectangle/Utilities/NotificationExtension.swift:20`), posted with a `Bool` by
  `Rectangle/PrefsWindow/ShortcutRecordingObserver.swift:61`. **Postable** by the spike to simulate
  recording. The custom manager subscribes and suspends firing while `true`.
- App-ignore / global-disable: `ApplicationToggle.shortcutsDisabled` is **`public private(set)
  static`** (`ApplicationToggle.swift:10`) — read-only externally. The only mutators are the private
  `disableShortcuts()`/`enableShortcuts()` (:50–68), reachable via the **public**
  `disableApp(appBundleId:)` / `enableApp(appBundleId:)` (:70–84) or the front-app workspace note
  (:94). **Decision:** `AppDelegate` passes its LIVE `applicationToggle` (and `WindowHistory`) into
  the spike runner (§3); the spike flips the real flag via the public `disableApp/enableApp`.
  **`disableApp` has compound side effects** (C2#disabledApps): it mutates the in-memory `disabledApps`
  set, writes `Defaults.disabledApps` via `saveDisabledApps`, and calls `disableShortcuts()` →
  `shortcutManager.unbindShortcuts()` + posts `windowSnapping(false)`; `enableApp` reverses these. So
  restoring the defaults value alone is NOT enough — see check 3's full teardown (initial-state
  precondition + `reloadFromDefaults()` + final-state assertion). The custom manager's fire path reads
  the public `ApplicationToggle.shortcutsDisabled` getter.

### 1.6 Conflict model — §3.3 / rev3 #3
- Precedent: `TodoManager.conflict(for:ignoringTodoDefaultsKey:userDefaults:) -> TodoShortcutConflict?`
  (`Rectangle/TodoMode/TodoManager.swift:284`), comparing `ShortcutCycle.ShortcutIdentity(shortcut)`
  against every `WindowAction.active` shortcut via
  `ShortcutCycle.shortcut(for:userDefaults:)` and the other todo keys. The `userDefaults` parameter
  is injectable (default `.standard`) — we exploit this to run conflict checks against an ISOLATED
  suite (C1#leak).
- **We mirror this**: reject a chord whose `ShortcutIdentity` matches any live `WindowAction`
  shortcut OR any other custom layout. The spike records the exact fields `ShortcutIdentity`
  compares (keyCode + modifierFlags) and asserts the comparison normalizes
  device-dependent/left-right modifier masks the same way MASShortcut does (C1#identityNormalize):
  if `ShortcutIdentity` does NOT normalize, the conflict check builds the identity from
  `MASShortcut.keyCode`/`.modifierFlags` so two recorder-equal chords compare equal.
- Window Actions use `PassthroughShortcutValidator` (`PrefsViewController.swift:145`) — Rectangle
  does NOT reject conflicts for its own actions, so OUR system owns the gate.

### 1.7 History / restore — §3.6 (C1#historyWriters)
- `AppDelegate.windowHistory: WindowHistory` (`AppDelegate.swift:15`). `WindowHistory` is two dicts:
  `restoreRects [CGWindowID: CGRect]` and `lastRectangleActions [CGWindowID: RectangleAction]`
  (`Rectangle/WindowHistory.swift`).
- **Known writers** (corrected — not only `WindowManager.execute`): `WindowManager` action
  execution, and **drag-snapping in `SnappingManager`** (`Rectangle/Snapping/SnappingManager.swift:261`
  and `:346`). The direct spike apply path (`setFrame`) calls **none** of these. Proof: snapshot the
  full contents of BOTH dicts before/after a custom snap and assert byte-for-byte unchanged.

### 1.8 WindowAction is a fixed enum (BLOCKER confirmed)
- `enum WindowAction: Int, Codable` + hardcoded `static let active` (`WindowAction.swift:13,141`);
  execution is notification-driven (`post()` → `Notification.Name(action.name)` →
  `WindowManager.execute`). Cannot add runtime cases ⇒ parallel subsystem required (as planned).

## 2. Canonical `HotkeyData` + serialization proof (§3.2 rev2 #3 / rev3 #4) (C1#serialization)
Rectangle already ships the exact Codable shape: `struct Shortcut: Codable { let keyCode: Int;
let modifierFlags: UInt }` with `init(masShortcut:)` / `toMASSHortcut()` (`WindowAction.swift:1182`).
On disk, MASShortcut persists via `MASDictionaryTransformer` as the dict
`{ "keyCode": Int, "modifierFlags": Int }` (lower-case keys), bound through
`MASDictionaryTransformerName` (`PrefsViewController.swift:115`, `ShortcutManager.swift:17`).

**Decision:** `HotkeyData` is a Codable struct `{ keyCode: Int, modifierFlags: UInt,
schemaVersion: Int = 1 }` mirroring `Shortcut`, with `init(_ MASShortcut)` and `toMASShortcut()`.
Persisted inside each `CustomLayout` in the JSON array under our own key
**`com.perg593.divvy2.customLayouts`**.

**Check-6 proof obligations (not just a programmatic round-trip):**
1. `MASDictionaryTransformer.reverseTransformedValue(MASShortcut)` produces EXACTLY
   `{ "keyCode": k, "modifierFlags": m }`, and `transformedValue(dict)` returns a MASShortcut equal
   (by keyCode+modifierFlags) to the original — i.e. our `HotkeyData` is interchangeable with the
   on-disk format Rectangle/`MASShortcutView` actually writes.
2. Bind a real `MASShortcutView` to a temp UserDefaults key (isolated suite), set a shortcut
   programmatically, and assert the emitted defaults value equals the same
   `{keyCode, modifierFlags}` dict — proving `HotkeyData` ⇄ recorder output, not just ⇄ a synthetic
   MASShortcut.
3. JSON round-trip `HotkeyData` and assert `keyCode`/`modifierFlags` preserved.
Commit the ONE representation + `schemaVersion`/migration note into `SPIKE.md`.

## 3. Harness design (minimal upstream coupling — R2)
- **New, isolated code** under `Rectangle/Divvy2Spike/`; the ONLY edit to existing files is one
  guarded call. Files:
  - `Divvy2SpikeRunner.swift` — orchestrates the 7 checks, writes `SPIKE.md`, logs `PASS`/`FAIL`
    and a final `SPIKE RESULT: PASS|FAIL (n/7)` line; sets a non-zero process exit code on any
    failure (CI-style gate).
  - `Divvy2SpikeCustomLayout.swift` — *minimal* spike types: `HotkeyData`, a `CustomLayout` stub,
    and `SpikeCustomLayoutShortcutManager` (register/unregister via `MASShortcutMonitor` honoring the
    §1.4 ownership rule; conflict check; lifecycle suspension). Spike-grade, NOT the final M1/M2 types.
- **Trigger (C1#coupling / C2#timing):** ONE guarded call placed at the END of
  `AppDelegate.accessibilityTrusted()` — AFTER `applicationToggle` (and the rest of the managers) are
  constructed (`AppDelegate.swift:135`), NOT in `applicationDidFinishLaunching` where
  `applicationToggle` is still nil until AX is trusted. Passes LIVE instances so no globals/private
  access are needed:
  `if CommandLine.arguments.contains("--divvy2-spike") { Divvy2SpikeRunner.run(applicationToggle:
  applicationToggle, windowHistory: AppDelegate.windowHistory) }`. If `applicationToggle` is somehow
  nil (AX not granted ⇒ `accessibilityTrusted()` never ran), the runner reports
  **environment-not-ready** and FAILS rather than proceeding. The spike code is compiled in but never
  reached on a normal launch (no shortcuts registered, no defaults touched, no window created).
  Optionally compile the runner body under `#if DEBUG` so Release cannot run it at all.
- **Invocation for the gate:** run the built binary directly so stdout is captured:
  `"…/Moving Van.app/Contents/MacOS/Moving Van" --divvy2-spike` (same bundle id/signature ⇒ keeps
  the AX grant). Output path: `$DIVVY2_SPIKE_OUT` (default repo root `SPIKE.md`); the path is logged.
- **Window-move target (C1#crossApp / C2#textedit / C3#helperApp): a DISPOSABLE purpose-built helper
  app is the GATING cross-app target — no third-party app (e.g. TextEdit) is ever touched.** We add a
  minimal second app target **`Divvy2SpikeHelper`** to the project: bundle id
  `com.perg593.divvy2.spikehelper`, a single resizable titled `NSWindow`, NO document model, and
  state-restoration explicitly OFF (`NSQuitAlwaysKeepsWindows=false`,
  `applicationSupportsSecureRestorableState` → false) so launching it can never restore prior windows
  and terminating it can never close user content. The spike launches a FRESH instance of THIS helper
  via `NSWorkspace.openApplication` (its built `.app`), captures the launched PID, and locates the
  window via `getFrontWindowElement()`. Before any move it VERIFIES the focused AX element's owning PID
  == the launched PID, bundle id == `com.perg593.divvy2.spikehelper`, and AX role == window —
  environment-not-ready FAIL on mismatch. Checks 4/5 move THAT window. Teardown terminates ONLY the
  launched helper PID. This gives real cross-app AX fidelity with zero user-state hazard. (A
  same-process own-`NSWindow` move is retained only as a NON-gating smoke; activation restored to
  `.accessory`.) Implementation note: adding the helper target is part of M0.5 build work and is the
  one structural project change this spike requires.
- **Isolation & crash-safety (C1#leak):**
  - Conflict checks (check 2) run against an **isolated `UserDefaults(suiteName: "com.perg593.divvy2.spike")`**
    — injected via the `userDefaults:` parameter — so the user's real prefs are never used for the
    synthetic WindowAction shortcut. The suite is removed (`removePersistentDomain`) in teardown.
  - For the lifecycle check that must touch the real `Defaults.disabledApps` (via `disableApp`), the
    spike snapshots the exact key value to a backup and funnels restore into ONE idempotent
    `cleanup()` invoked from `defer`, `atexit`, AND a `DispatchSourceSignal` (SIGINT/SIGTERM) on the
    MAIN queue (C3#signal — raw POSIX handlers can't safely call Foundation/Defaults). `cleanup()`
    re-enables (`enableApp`), restores `Defaults.disabledApps`, and calls `reloadFromDefaults()` so
    Rectangle's bindings are rebound and the in-memory set is in sync even on early exit.

### 3.1 Implementation-precision notes (folded in from Codex round-4 GREEN)
- **Helper launch/acquisition (bounded poll, avoid false-FAIL).** Launch with
  `NSWorkspace.OpenConfiguration` where `activates = true` and `createsNewApplicationInstance = true`
  (force a fresh instance, never reuse a stale helper). Then a bounded poll (e.g. ≤3 s, 50 ms steps):
  wait until `NSWorkspace.shared.frontmostApplication?.processIdentifier == launchedPID`, THEN wait
  until `getFrontWindowElement()` returns the helper window (PID/bundle/role verified per §3) — only
  then proceed; on timeout report environment-not-ready.
- **Helper product discovery.** The main scheme builds `Divvy2SpikeHelper` as a **target dependency**;
  resolve its URL as `Bundle.main.bundleURL.deletingLastPathComponent()/Divvy2SpikeHelper.app` and
  assert existence before launch (don't search/guess paths).
- **DispatchSourceSignal lifetime.** Retain the signal source for the process lifetime and pair it
  with `signal(SIGINT, SIG_IGN)` / `signal(SIGTERM, SIG_IGN)` so the dispatch handler (not the default
  disposition) actually runs `cleanup()`.

## 4. The seven gated checks (each → real API + INDEPENDENT assertion in SPIKE.md)
1. **Register + reload + ownership (rev2 #1 / C1#ownership / C3#nonMutating).** Via the conflict-gated
   manager, register two novel chords on `MASShortcutMonitor.shared()` with **counter closures** (the
   real apply path is exercised in checks 4/5; check 1 stays non-mutating). Assert both
   `registerShortcut` returned `true` and the manager tracks exactly those two owned chords. Invoke
   each stored closure directly → assert exactly one increment each (wiring + no double-invoke).
   `unregisterShortcut` one, then re-register it from a simulated store-change; assert tracking is
   correct (single entry, no leak). **Non-mutating ownership guard (no Rectangle action is fired):**
   pre-register a SPIKE-OWNED stand-in chord on the same shared monitor with our own counter closure
   (a proxy for "an existing binding" — never a real Rectangle action); run a rejected duplicate of it
   through the manager (check 2); then assert (a) the manager recorded ZERO `register`/`unregister`
   calls for that rejected chord, and (b) the stand-in is still registered and its counter closure
   still fires when invoked — proving the manager cannot tear down a chord it does not own. Unregister
   the stand-in in teardown.
2. **Conflict rejection (rev3 #3).** In the isolated suite, inject a `WindowAction` shortcut
   (e.g. `leftHalf` = ⌃⌥←) via `MASDictionaryTransformer`; attempt a custom layout with the SAME
   chord → assert **rejected** (matches a live WindowAction) AND that NO `registerShortcut`/
   `unregisterShortcut` call was made for it. Duplicate **custom** chord → **rejected**, no monitor
   mutation. Non-conflicting chord → **registered** (`registerShortcut == true`). Record each Bool.
3. **Lifecycle suppression (rev3 #1 / C2#disabledApps).** *Recording:* post
   `Notification.Name.shortcutRecording`(`true`) → assert a fired custom chord does NOT execute; post
   `false` → resumes. *Disabled-state* (only valid from an initially ENABLED state — if
   `ApplicationToggle.shortcutsDisabled` is already `true` at entry, record **environment-not-ready**
   and FAIL this sub-check): capture `initial = ApplicationToggle.shortcutsDisabled` (must be `false`)
   and snapshot `Defaults.disabledApps`; call `applicationToggle.disableApp(appBundleId: <unique fake
   id>)` → assert the REAL flag is now `true` and the custom chord is suppressed; then
   `applicationToggle.enableApp(<same fake id>)`, restore `Defaults.disabledApps`, call
   `applicationToggle.reloadFromDefaults()`, and assert the flag returned to `initial` (`false`) and
   the in-memory set is back in sync. The teardown is also wired into `defer`/`atexit`/signal handlers.
4. **Real cross-app window move + no history writes (rev2 #2).** Snapshot the full contents of
   `windowHistory.restoreRects` and `.lastRectangleActions`; move the **helper-app** window
   (`Divvy2SpikeHelper`, §3) via `getFrontWindowElement().setFrame(target.screenFlipped)`; after
   settle, assert read-back `frame` matches target (≤1px) AND both history dicts are byte-for-byte
   unchanged. Repeat the same chord twice → idempotent (no drift).
5. **Coordinate contract with INDEPENDENT ground truth (rev3 #2 / C1#circular).** For full-ish,
   **top-half**, **bottom-half**, and secondary display (if attached): map a `NormalizedRect` onto
   `currentScreen.visibleFrame` → AppKit target. Apply via `setFrame(target.screenFlipped)`. Assert
   THREE independent ways: (a) AX read-back `frame` equals an **expected AX rect computed WITHOUT
   `screenFlipped`** — `expected.origin.y = NSScreen.screens[0].frame.maxY - target.maxY`, x/size
   direct; (b) `CGWindowListCopyWindowInfo` bounds (top-left global coords) for the window equal the
   AppKit target converted by the same explicit formula; (c) top-half vs bottom-half AX `origin.y`
   differ in the correct direction (sign check catching a flipped axis). A bug in `screenFlipped`
   fails (a)/(b) instead of hiding. **Notched/secondary fallback (rev3 #10):** if absent, assert
   menu-bar/Dock `visibleFrame` behavior on the primary display and record the unavailable-hardware
   case; the gate still passes.
6. **Hotkey serialization (rev2 #3 / rev3 #4 / C1#serialization).** Execute the three proof
   obligations in §2 (MASDictionaryTransformer exact-dict; `MASShortcutView`→defaults emission in an
   isolated suite; `HotkeyData` JSON round-trip). Commit the representation + migration note to
   `SPIKE.md`.
7. **API confirmation (finding #7).** Record into `SPIKE.md` the confirmed names/signatures for
   `WindowAction`, `WindowManager`/`WindowCalculation*` (bypassed), `ScreenDetection`,
   `AccessibilityElement`, `ShortcutManager`/`MASShortcutBinder` vs `MASShortcutMonitor`,
   `ApplicationToggle`, `PrefsViewController` + the real `MASShortcutView` recorder, `WindowHistory`,
   and the `Shortcut` Codable — as enumerated in §1.

**Gate:** all seven PASS (check 5's secondary-display sub-case may be recorded as unavailable
without failing). Final line: `SPIKE RESULT: PASS|FAIL (n/7)`.

## 5. Risks / unknowns to resolve during the spike
- **R-spike-1 Cross-app AX fidelity (now gating).** The disposable `Divvy2SpikeHelper` move is the real
  cross-app proof (separate process, real AX, but fully owned and document-free); mitigate AX settle
  with a bounded read-back retry and ≤1px tolerance. Helper-not-obtainable ⇒ environment-not-ready
  FAIL, not pass.
- **R-spike-2 LSUIElement activation.** The non-gating own-window smoke briefly flips OUR app's
  activation to `.regular` (possible transient Dock icon); restored to `.accessory`. The gating path
  uses the separate helper-app process, so no activation flip of our own app is needed there.
- **R-spike-7 Helper-app target cost.** Adding the `Divvy2SpikeHelper` app target is the one
  structural project change M0.5 requires; it is a throwaway de-risking target (kept out of Release
  distribution) and can be removed post-spike if desired.
- **R-spike-3 MASShortcutMonitor coexistence + chord-scoped unregister (C1#ownership).** Register only
  novel chords; track owned chords; never unregister non-owned/rejected chords; assert Rectangle's
  bindings still fire post-teardown.
- **R-spike-4 Real-prefs safety (C1#leak / C2#disabledApps).** Conflict checks use an isolated suite. The
  one real-defaults touch (`disableApp`) runs only from an initially-enabled state with a unique fake
  bundle id, and teardown does `enableApp(fake)` + restore `Defaults.disabledApps` +
  `reloadFromDefaults()` + assert the flag returned to its initial value — all wired into
  `defer`/`atexit`/signal handlers so an early exit still restores Rectangle's bindings.
- **R-spike-5 Writing SPIKE.md.** App is unsandboxed; write to `$DIVVY2_SPIKE_OUT` (repo-root default)
  and log the path.
- **R-spike-6 ShortcutIdentity normalization (C1#identityNormalize).** Confirm `ShortcutCycle.ShortcutIdentity`
  equality matches recorder-equal chords (modifier mask normalization); if not, build identities from
  `MASShortcut.keyCode`/`.modifierFlags` directly so the conflict check can't miss a real duplicate.

## 7. Review gate
This design goes to adversarial Codex review (`codex exec -s read-only`) before any spike code. Target:
zero BLOCKER/MAJOR. After the spike runs GREEN and `SPIKE.md` is produced, a second Codex checkpoint
reviews the spike RESULTS before M1.
